## 1.1 왜 첫 장에서 "기술 이름"보다 "흐름"을 먼저 봐야 하는가

데이터 사이언스를 처음 배우면 알고리즘 이름이 한꺼번에 쏟아진다.

- 회귀
- 의사결정나무
- 랜덤 포레스트
- 신경망
- Transformer
- LLM

이름만 보면 전부 서로 다른 기술처럼 보이지만, 실제로는 하나의 긴 흐름 안에 있다. 첫 장의 역할은 이 흐름을 지도처럼 보여 주는 것이다.

쉽게 비유하면 이렇다.

- 통계학은 자와 저울이다
- 머신러닝은 분류와 예측을 잘하는 도구 상자다
- 딥러닝은 복잡한 특징을 스스로 읽는 엔진이다
- LLM은 설명, 요약, 생성, 검색 보조까지 겸하는 범용 도구다

즉, 첫 장에서는 "무엇이 최신인가"보다 **어떤 문제가 어떤 기술을 불러왔는가**를 보는 것이 더 중요하다.

## 1.2 데이터 사이언스는 어떻게 발전했는가

데이터 사이언스는 갑자기 생긴 분야가 아니다. 통계학, 데이터마이닝, 머신러닝, 딥러닝, 생성형 AI가 층층이 쌓이면서 만들어진 결과다.

### 1.2.1 통계학의 시대: 먼저 가설을 세우고 검증하다

현대 데이터 분석의 뿌리는 통계학이다.  
통계학의 핵심 질문은 대체로 다음과 같았다.

- 두 집단은 정말 차이가 있는가
- 이 관계는 우연인가 아닌가
- 이 계수는 얼마나 신뢰할 수 있는가

즉, 통계학은 "가설을 세우고 그것이 맞는지 검증하는 체계"를 발전시켰다.

이 접근은 매우 중요하다. 지금도 회귀분석, 가설검정, 신뢰구간 같은 개념은 거의 모든 데이터 분석의 뼈대를 이룬다.

하지만 통계학 중심 접근에는 한계도 있었다.

- 데이터가 아주 커지면 다루기 어렵다
- 복잡한 비선형 관계를 표현하기 어렵다
- 예측 성능을 극대화하는 데는 한계가 있다

### 1.2.2 데이터마이닝의 시대: 패턴을 먼저 찾기 시작하다

컴퓨터 자원이 늘어나면서 사람들은 더 이상 "가설만 검증"하지 않고, 데이터에서 패턴을 직접 찾기 시작했다. 이것이 데이터마이닝 시대다.

데이터마이닝의 감각은 다음에 가깝다.

- 어떤 고객들이 함께 상품을 사는가
- 어떤 문서들이 비슷한가
- 어떤 집단이 숨어 있는가

통계학이 질문을 먼저 만들고 답을 찾는 방식이라면, 데이터마이닝은 **데이터를 먼저 들여다보며 구조를 발견하는 방식**에 가깝다.

### 1.2.3 머신러닝의 시대: 예측 성능이 중심이 되다

2000년대 이후 머신러닝은 데이터 사이언스의 중심으로 올라섰다. 이 시기 핵심 질문은 훨씬 실용적이다.

- 얼마나 잘 맞히는가
- 어떤 모델이 더 정확한가
- 어떤 특성을 넣으면 성능이 좋아지는가

머신러닝 시대의 핵심 특징은 **특성 공학**이다.  
즉, 좋은 모델만큼이나 좋은 입력 표현을 만드는 능력이 중요했다.

랜덤 포레스트, SVM, XGBoost 같은 모델들이 이 시기를 대표한다.

### 1.2.4 딥러닝의 시대: 특성을 사람이 덜 만들게 되다

딥러닝의 가장 큰 변화는 사람이 직접 특성을 많이 설계하지 않아도 된다는 점이다.  
이미지, 음성, 텍스트처럼 복잡한 데이터에서 모델이 표현을 스스로 학습하기 시작했다.

예를 들어 이미지 분류를 생각해 보자.

- 전통 방식: 모서리, 색, 질감 같은 특징을 사람이 설계
- 딥러닝: 원시 픽셀에서 유용한 표현을 스스로 학습

이 변화는 매우 컸다. 하지만 동시에 더 많은 데이터와 연산 자원을 요구하게 되었다.

### 1.2.5 LLM의 시대: 언어를 다루는 범용 도구의 등장

최근의 가장 큰 변화는 대규모 언어 모델(LLM)이다.  
LLM은 단순한 분류 모델이 아니라, 요약, 질의응답, 코드 생성, 검색 보조, 문서 해석까지 한꺼번에 수행할 수 있다.

이 때문에 LLM은 데이터 분석에서도 새로운 역할을 맡게 되었다.

- EDA 초안 생성
- 코드 초안 작성
- 문서 요약
- 변수 설명 정리
- 유사 사례 탐색

하지만 여기서 반드시 기억해야 할 점이 있다.  
LLM은 매우 유용하지만, **항상 사실만 말하는 분석기계는 아니다.**

<표 1-1: 데이터 사이언스 패러다임의 변화>

| 시대 | 핵심 접근 | 대표 기법 | 강점 | 주요 한계 |
| ---- | --------- | --------- | ---- | --------- |
| 통계학 | 가설 검증 | 회귀, t-test, ANOVA | 해석이 명확함 | 복잡한 패턴 처리 한계 |
| 데이터마이닝 | 패턴 발견 | 연관규칙, 군집 분석 | 구조 탐색에 강함 | 비정형 데이터 한계 |
| 머신러닝 | 예측 모델링 | SVM, Random Forest, XGBoost | 정형 데이터 성능 우수 | 특성 설계 부담 |
| 딥러닝 | 표현 학습 | CNN, RNN, Transformer | 비정형 데이터 강함 | 데이터/연산 자원 요구 큼 |
| LLM | 범용 언어·지식 활용 | GPT, Claude, Gemini | 요약·생성·보조 분석 강함 | 환각, 비용, 검증 필요 |

### 1.2.6 AI가 분석가를 대체하는가

이 질문은 첫 장에서 반드시 다루어야 한다.  
결론부터 말하면, AI는 분석가의 일부 업무를 대체할 수 있지만 분석가 전체를 바로 대체한다고 보기는 어렵다.

AI가 잘하는 일:

- 반복적 코드 작성
- 기초 통계 요약
- 문서 정리
- 시각화 초안 생성
- 기본 설명 초안 작성

AI가 여전히 어려워하는 일:

- 문제 정의
- 이해관계자 조율
- 잘못된 데이터나 맥락 오류 판단
- 윤리적 판단
- "이 분석이 실제로 쓸모 있는가"를 결정하는 일

즉, 현실적인 그림은 "분석가 대 AI"가 아니라 **AI를 잘 활용하는 분석가**의 등장이다.

## 1.3 머신러닝의 네 가지 학습 방식

초보자는 알고리즘 이름을 외우다가 금방 혼란스러워진다. 이럴 때는 알고리즘보다 먼저 **학습 방식의 종류**를 구분하는 편이 좋다.

### 1.3.1 지도학습: 정답이 있는 학습

지도학습은 입력과 정답이 함께 있을 때 학습한다.

예를 들면:

- 고객 이탈 여부 예측
- 집값 예측
- 스팸 메일 분류

비유하면 지도학습은 **정답지가 있는 문제집**을 푸는 것과 같다.

### 1.3.2 비지도학습: 정답 없이 구조를 찾는 학습

비지도학습은 정답 레이블이 없을 때 데이터 안에 숨어 있는 구조를 찾는다.

예를 들면:

- 고객 세분화
- 문서 군집화
- 이상 패턴 탐지

비유하면 비지도학습은 **정답 없이 비슷한 것끼리 스스로 묶는 작업**이다.

### 1.3.3 강화학습: 시행착오를 통해 보상을 극대화하는 학습

강화학습은 정답을 바로 받지 않고, 행동의 결과로 보상을 받으며 학습한다.

예를 들면:

- 게임 플레이
- 추천 정책 최적화
- 재고나 가격 결정

비유하면 강화학습은 **직접 해 보면서 점수를 가장 많이 따는 전략을 배우는 방식**이다.

### 1.3.4 자기지도학습: 데이터 안에서 스스로 문제를 만드는 학습

자기지도학습은 레이블이 없는 데이터에서 학습 신호를 스스로 만든다.  
오늘날 LLM의 핵심 학습 방식도 여기에 속한다.

예를 들어:

- 다음 단어 예측
- 가려진 단어 맞히기
- 문장 임베딩 사전학습

비유하면 자기지도학습은 **문제집이 없는 상태에서 스스로 빈칸 문제를 만들어 푸는 것**과 비슷하다.

<표 1-2: 머신러닝 학습 방식 비교>

| 유형 | 학습 데이터 | 핵심 질문 | 대표 예시 |
| ---- | ----------- | --------- | --------- |
| 지도학습 | 입력 + 정답 | 무엇을 예측할까 | 분류, 회귀 |
| 비지도학습 | 입력만 | 어떤 구조가 숨어 있나 | 군집화, 차원 축소 |
| 강화학습 | 상태, 행동, 보상 | 어떤 행동이 장기적으로 유리한가 | 게임, 정책 최적화 |
| 자기지도학습 | 입력만 | 데이터 안에서 무엇을 스스로 배울 수 있나 | LLM, 임베딩 학습 |

### 1.3.5 왜 이 구분이 중요한가

같은 데이터라도 질문이 달라지면 학습 방식이 달라진다.

예를 들어 고객 로그 데이터가 있다고 하자.

- 이탈 여부를 맞히면 지도학습
- 고객군을 나누면 비지도학습
- 어떤 쿠폰을 보내야 장기 가치가 커지는지 배우면 강화학습
- 텍스트 로그 표현을 미리 학습하면 자기지도학습

즉, 좋은 분석가는 알고리즘 이름보다 먼저 **문제 구조를 분류할 수 있어야 한다.**

## 1.4 데이터 분석 워크플로우: 전통적 방식과 AI 증강 방식

데이터 분석은 모델 하나만 돌린다고 끝나는 일이 아니다.  
문제 정의, 데이터 확인, 전처리, 모델링, 평가, 보고, 배포까지 이어지는 전체 흐름이 중요하다.

### 1.4.1 CRISP-DM: 전통적인 분석 절차

오랫동안 가장 널리 쓰인 데이터 분석 프레임워크는 CRISP-DM이다.

기본 구조는 다음과 같다.

1. 비즈니스 이해
2. 데이터 이해
3. 데이터 준비
4. 모델링
5. 평가
6. 배포

이 절차가 여전히 중요한 이유는, 분석을 무조건 모델 중심으로 밀어붙이지 않고 **문제 정의와 검증**을 앞단에 두기 때문이다.

### 1.4.2 AI 증강 워크플로우

LLM과 AutoML이 들어오면서 이 절차는 조금씩 바뀌고 있다.  
단계가 사라지는 것은 아니지만, 각 단계에서 사람이 하는 일과 AI가 도와주는 일이 달라진다.

예를 들면:

- 비즈니스 이해: LLM이 유사 사례 요약 보조
- 데이터 이해: 자동 프로파일링과 자연어 요약 보조
- 모델링: AutoML이 빠른 기준선 제공
- 평가/배포: 실험 추적과 운영 자동화 강화

즉, AI는 단계를 없애는 것이 아니라 **각 단계의 반복 업무를 줄이는 방향**으로 작동한다.

<표 1-3: 전통 워크플로우와 AI 증강 워크플로우 비교>

| 단계 | 전통적 방식 | AI 증강 방식 |
| ---- | ----------- | ------------ |
| 비즈니스 이해 | 수동 조사와 회의 중심 | LLM 기반 자료 요약 보조 |
| 데이터 이해 | 수동 EDA 코드 작성 | 자동 프로파일링, 자연어 해석 보조 |
| 데이터 준비 | 수작업 비중 큼 | 코드 생성/자동화 도구 활용 |
| 모델링 | 수동 알고리즘 선택과 튜닝 | AutoML로 기준선 신속 구축 |
| 평가 | 수동 비교와 문서화 | 실험 추적과 비교 자동화 |
| 배포 | 수동 스크립트 중심 | MLOps 파이프라인 활용 |

### 1.4.3 여기서 중요한 원칙

AI 도구가 들어와도 바뀌지 않는 원칙이 있다.

1. 문제 정의는 사람이 책임진다
2. 데이터 품질 검증은 사람이 최종 판단한다
3. 모델 결과의 의미 해석은 사람이 한다
4. 자동 생성 결과는 검증 없이 그대로 쓰지 않는다

즉, AI 증강 워크플로우는 자동화의 확대이지, **책임의 자동화**가 아니다.

## 1.5 LLM 기반 탐색적 데이터 분석(EDA)

이 장의 실습은 LLM 기반 EDA다.  
따라서 개념 설명도 실습과 붙여서 읽는 것이 중요하다.

### 1.5.1 전통적 EDA의 한계

탐색적 데이터 분석은 거의 모든 프로젝트의 출발점이다. 그런데 실제로는 다음 같은 반복 작업이 많다.

- 기본 통계량 확인
- 결측치 비율 확인
- 분포 시각화
- 상관관계 확인
- 이상치 여부 탐색

숙련된 분석가에게도 이 작업은 시간이 걸린다. 특히 새로운 데이터셋을 계속 다뤄야 할 때 반복 부담이 크다.

### 1.5.2 LLM 기반 EDA는 무엇을 바꾸는가

LLM 기반 EDA는 크게 두 가지 방식으로 생각할 수 있다.

1. 대화형 분석
2. API 기반 분석

대화형 방식은 빠르지만 재현성이 약할 수 있고, API 기반 방식은 재현성과 자동화에 유리하다.

이 저장소의 실습은 두 번째 감각에 더 가깝다.  
즉, 데이터 프로파일을 만들고, 그 요약을 LLM에 전달해 분석 초안을 받는 흐름이다.

### 1.5.3 실습: LLM 기반 자동 EDA

[practice/chapter01/code/1-4-llm-eda.py](/Users/callii/Documents/dataScience/practice/chapter01/code/1-4-llm-eda.py) 는 데이터 프로파일 생성, 시각화, LLM 해석 요청을 한 흐름으로 보여 준다.

코드의 핵심 구조는 다음과 같다.

```python
profile = generate_data_profile(df, dataset_name)
saved_files = create_eda_visualizations(df, dataset_name)
insights = generate_llm_insights(profile, use_api=True)
```

이 세 줄이 말해 주는 것은 분명하다.

1. 먼저 사람이 읽을 수 있는 구조화된 프로파일을 만든다
2. 시각 자료를 만든다
3. 그 요약 정보를 LLM에 전달해 해석 초안을 받는다

즉, LLM은 데이터를 직접 "마법처럼 이해"하는 것이 아니라, **정리된 입력을 바탕으로 해석을 보조**하는 것이다.

### 1.5.4 이 실습에서 꼭 봐야 할 것

이 실습은 단순히 "Claude API를 불렀다"는 데 의미가 있지 않다.  
오히려 다음이 더 중요하다.

1. 어떤 정보를 입력으로 전달했는가
2. LLM이 무엇을 잘 요약했는가
3. 무엇을 과도하게 해석할 위험이 있는가
4. 결과를 재현 가능한 형태로 저장했는가

예를 들어 코드에는 프로파일 생성 함수가 있다.

```python
profile = {
    "dataset_name": dataset_name,
    "shape": {"rows": int(df.shape[0]), "columns": int(df.shape[1])},
    "columns": {},
    "missing_values": {},
    "correlations": {}
}
```

이 구조를 보면 실습의 교육 포인트가 분명하다.  
LLM에게 원본 데이터를 그대로 던지는 것이 아니라, **핵심 통계를 정리해 전달하는 방식**이다.

### 1.5.5 전통적 EDA와 LLM 기반 EDA 비교

<표 1-4: 전통적 EDA와 LLM 기반 EDA 비교>

| 항목 | 전통적 EDA | LLM 기반 EDA |
| ---- | ---------- | ------------- |
| 속도 | 상대적으로 느림 | 빠른 초안 생성 가능 |
| 초기 인사이트 | 분석가 경험 의존 | 자동 요약 가능 |
| 재현성 | 코드화하면 높음 | 세션형 도구는 약할 수 있음 |
| 해석 신뢰성 | 사람이 직접 검토 | 검증 없는 사용은 위험 |
| 비용 | 인력 시간 중심 | API 비용 발생 가능 |

핵심은 대체 관계가 아니라 **보완 관계**라는 점이다.  
LLM은 첫 탐색을 빠르게 해 주지만, 최종 판단은 전통적 검증 절차로 마무리해야 한다.

### 1.5.6 LLM EDA의 주의점

LLM 기반 EDA를 쓸 때는 세 가지를 항상 점검해야 한다.

1. **환각**: 데이터에 없는 패턴을 그럴듯하게 말할 수 있다
2. **보안**: 민감 데이터를 외부 API로 보내는 문제
3. **비용**: 대규모 데이터는 호출 비용이 커질 수 있다

즉, LLM은 유용하지만 무비판적으로 신뢰하면 안 된다.

## 1.6 전통 ML과 딥러닝은 어떻게 선택할 것인가

첫 장의 마지막에서는 학생이 자주 묻는 질문에 답해야 한다.

"그래서 결국 전통 ML과 딥러닝 중 무엇을 써야 하나요?"

### 1.6.1 정형 데이터에서는 왜 전통 ML이 강한가

정형 데이터에서는 트리 기반 모델이 매우 강한 경우가 많다.

- 변수 단위가 섞여 있어도 잘 작동
- 비선형 관계를 비교적 잘 다룸
- 적은 데이터에서도 안정적
- 해석 도구를 붙이기 쉬움

그래서 테이블 데이터 문제에서는 여전히 XGBoost, LightGBM, Random Forest가 강력한 기준선이다.

### 1.6.2 딥러닝이 유리한 상황

딥러닝은 다음 조건에서 특히 강하다.

- 이미지, 텍스트, 음성 등 비정형 데이터
- 데이터 양이 충분히 많음
- 표현 학습이 핵심
- 여러 모달리티를 함께 다룸
- 사전학습 모델을 활용할 수 있음

즉, 딥러닝은 만능이라서가 아니라 **적합한 문제에서 강력하다**고 이해해야 한다.

### 1.6.3 선택 기준 표

<표 1-5: 전통 ML vs 딥러닝 선택 기준>

| 고려 요소 | 전통 ML 유리 | 딥러닝 유리 |
| --------- | ------------ | ----------- |
| 데이터 유형 | 정형 데이터 | 이미지, 텍스트, 음성, 시계열 |
| 데이터 규모 | 중소규모 | 대규모 |
| 해석 필요성 | 높음 | 낮거나 보완 가능 |
| 개발 속도 | 빠름 | 상대적으로 느림 |
| 컴퓨팅 자원 | CPU 중심 | GPU 권장 |

### 1.6.4 실무적인 접근

가장 현실적인 접근은 다음 순서다.

1. 간단하고 강한 기준선부터 만든다
2. 요구 성능이 충분한지 본다
3. 부족하면 더 복잡한 모델로 확장한다

즉, 처음부터 가장 화려한 모델을 고르는 것이 아니라, **가장 합리적인 출발점**을 잡는 것이 중요하다.
