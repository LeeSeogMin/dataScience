## 2.1 전처리는 "귀찮은 준비"가 아니라 모델의 시야를 만드는 과정이다

머신러닝을 처음 배울 때 가장 흔한 오해는 "좋은 모델만 고르면 성능은 알아서 나온다"는 생각이다. 하지만 실제로는 입력 데이터가 잘못 정리되어 있으면 모델은 쉽게 흔들린다.

쉽게 비유하면 전처리는 안경을 닦는 일과 같다.

- 안경이 더러우면 세상이 흐리게 보인다
- 아무리 똑똑한 사람이 써도 흐린 렌즈로는 정확히 보기 어렵다

데이터도 마찬가지다.

- 결측치가 섞여 있거나
- 단위가 뒤죽박죽이거나
- 필요 없는 변수가 너무 많거나
- 의미 있는 패턴이 드러나지 않으면

모델은 제대로 학습하기 어렵다.

즉, 전처리는 모델을 "꾸미는 과정"이 아니라, **모델이 무엇을 보고 무엇을 중요하게 여길지 결정하는 과정**이다.

<표 2-1: 전처리가 실제로 바꾸는 것>

| 전처리 문제 | 그대로 둘 때 생기는 일 | 전처리의 역할 |
| ----------- | ---------------------- | ------------- |
| 결측치 | 학습 불가 또는 편향 | 정보 손실과 왜곡 최소화 |
| 스케일 차이 | 특정 변수가 거리 계산을 지배 | 변수 간 영향 균형 조정 |
| 고차원 입력 | 계산 증가, 과적합 위험 증가 | 핵심 구조만 남기기 |
| 표현 부족 | 의미를 잘 못 배움 | 모델이 배우기 쉬운 입력 설계 |

이 장의 핵심은 전처리를 "보조 작업"이 아니라 **모델링의 시작점**으로 보는 데 있다.

## 2.2 결측치: 비어 있는 칸이 아니라 데이터 생성 과정의 흔적

현실 데이터에는 거의 항상 결측치가 있다.

- 설문 응답 거부
- 센서 오류
- 시스템 누락
- 데이터 입력 실수
- 일부러 숨긴 정보

문제는 결측치를 어떻게 처리하느냐에 따라 결과가 크게 달라질 수 있다는 점이다.

### 2.2.1 모델은 왜 결측치를 싫어할까

많은 알고리즘은 결측치를 직접 다루지 못한다.

- 선형 회귀는 값을 계산하려면 숫자가 필요하다
- KNN은 거리 계산을 하려면 모든 좌표가 필요하다
- 신경망도 입력값이 비어 있으면 가중합 계산이 불가능하다

그래서 결측치를 만나면 보통 두 가지 중 하나를 해야 한다.

1. 결측이 있는 샘플을 제거한다
2. 어떤 값으로 대체한다

둘 다 비용이 있다.

- 제거하면 정보가 줄어든다
- 대체하면 인위적인 패턴이 들어간다

즉, 결측치 처리에는 항상 트레이드오프가 있다.

### 2.2.2 결측의 세 가지 유형

결측은 단순히 "비어 있음"으로 끝나지 않는다. **왜 비었는가**가 중요하다.

가장 직관적인 비유는 시험 답안지다.

- **MCAR**: 답안지가 우연히 찢어져 일부가 사라짐
- **MAR**: 특정 반 학생들만 더 많이 빠뜨림
- **MNAR**: 어려운 문항이라 일부러 비워 둠

데이터 문제에 대응시키면 다음과 같다.

<표 2-2: 결측 유형의 직관적 구분>

| 유형 | 뜻 | 직관적 예시 |
| ---- | --- | ------------ |
| MCAR | 아무 변수와도 무관하게 빠짐 | 파일 손상, 시스템 오류 |
| MAR | 다른 관측 변수와 관련되어 빠짐 | 고소득층이 소득 공개를 꺼림 |
| MNAR | 빠진 값 자체와 관련되어 빠짐 | 소득이 낮을수록 소득을 숨김 |

이 구분이 중요한 이유는 처리 전략이 달라지기 때문이다.

- MCAR이면 행 삭제가 덜 위험할 수 있다
- MAR이면 다른 변수 정보를 활용한 대체가 유리하다
- MNAR이면 단순 대체로는 편향을 없애기 어렵다

### 2.2.3 결측치 처리 전략 비교

대표적인 처리 전략은 다음 네 가지다.

1. **행 삭제**
2. **단순 대체**
3. **모델 기반 대체**
4. **결측 지시 변수 추가**

<표 2-3: 결측치 처리 전략 비교>

| 방법 | 장점 | 단점 | 적합한 상황 |
| ---- | ---- | ---- | ----------- |
| 행 삭제 | 단순함 | 정보 손실 큼 | 결측 비율이 매우 낮을 때 |
| 평균/중앙값 대체 | 빠르고 쉬움 | 분산 왜곡 가능 | 빠른 기준선 필요 |
| KNN/반복 대체 | 관계를 더 잘 반영 | 계산 비용 큼 | 정확도 중요, 변수 간 상관 있음 |
| 결측 지시 변수 | 결측 자체를 정보로 활용 | 차원 증가 | 결측 여부가 의미 있을 때 |

여기서 가장 중요한 포인트는 "결측치는 무조건 채우면 된다"가 아니라는 점이다.  
결측 자체가 중요한 신호라면, 결측 여부를 따로 변수로 남기는 것이 더 나을 수 있다.

### 2.2.4 실습: 결측치 처리 방법 비교

[practice/chapter02/code/2-1-missing-values.py](/Users/callii/Documents/dataScience/practice/chapter02/code/2-1-missing-values.py) 는 결측치 처리 방법을 비교하는 예시다.

이 실습은 California Housing 데이터를 사용하고, 인위적으로 결측을 넣은 뒤 다음 방법을 비교한다.

- 행 삭제
- 평균 대체
- 중앙값 대체
- KNN 대체
- 반복 대체(MICE)

실습 코드의 핵심 구조는 아래와 같다.

```python
methods = [
    ("행 삭제", None),
    ("평균 대체", SimpleImputer(strategy='mean')),
    ("중앙값 대체", SimpleImputer(strategy='median')),
    ("KNN 대체 (K=5)", KNNImputer(n_neighbors=5)),
    ("반복 대체 (MICE)", IterativeImputer(max_iter=10, random_state=42))
]
```

이 부분에서 읽어야 할 핵심은 "정답이 하나가 아니다"는 점이다.

- 가장 빠른 방법이 항상 가장 좋은 방법은 아니다
- 가장 정교한 방법이 항상 실무 최선도 아니다

결과를 해석할 때는 보통 아래를 같이 본다.

1. RMSE가 얼마나 줄었는가
2. 처리 시간이 얼마나 늘었는가
3. 결측 비율이 실제 현장에서도 이 정도인가
4. 운영 환경에서 계산 비용을 감당할 수 있는가

즉, 결측치 처리는 **성능과 비용의 균형 문제**다.

## 2.3 스케일링: 숫자의 크기를 맞춰 공정하게 비교하기

어떤 모델은 변수의 크기 차이에 매우 민감하다.  
키(cm)와 연봉(만원)을 같은 테이블에 두면, 숫자 크기가 큰 연봉 변수가 거리 계산을 거의 지배하게 된다.

쉽게 말하면, 어떤 변수는 너무 큰 목소리로 말하고 다른 변수는 거의 들리지 않게 되는 셈이다.

### 2.3.1 어떤 알고리즘이 스케일링에 민감한가

거리 기반 알고리즘은 스케일링에 민감하다.

- KNN
- K-Means
- SVM
- DBSCAN

이 알고리즘들은 "얼마나 가까운가"를 계산하기 때문에, 범위가 큰 변수가 계산을 지배하면 결과가 왜곡된다.

반면 트리 기반 모델은 비교적 덜 민감하다.

- 의사결정나무
- 랜덤 포레스트
- XGBoost
- LightGBM

이들은 값을 직접 거리로 보지 않고, 분할 기준으로 나누기 때문에 스케일 변화의 영향을 거의 받지 않는다.

<표 2-4: 스케일링 민감도 비교>

| 모델 계열 | 스케일링 필요성 | 이유 |
| --------- | --------------- | ---- |
| KNN, SVM, K-Means | 높음 | 거리 계산에 직접 영향 |
| 신경망 | 높음 | 학습 안정성과 수렴 속도에 영향 |
| 선형 회귀/로지스틱 회귀 | 보통~높음 | 계수 크기와 최적화 안정성에 영향 |
| 트리 기반 모델 | 낮음 | 분할 기준 중심이라 거리 영향 적음 |

### 2.3.2 대표적인 스케일링 방법

가장 많이 쓰는 방법은 네 가지다.

- StandardScaler
- MinMaxScaler
- RobustScaler
- MaxAbsScaler

<표 2-5: 스케일링 방법 비교>

| 방법 | 직관 | 장점 | 주의점 |
| ---- | ---- | ---- | ------ |
| StandardScaler | 평균 0, 표준편차 1로 맞춤 | 가장 범용적 | 이상치에 민감 |
| MinMaxScaler | 일정 범위로 압축 | 신경망 입력에 유용 | 이상치에 매우 민감 |
| RobustScaler | 중앙값과 IQR 사용 | 이상치에 강함 | 해석이 덜 직관적일 수 있음 |
| MaxAbsScaler | 최대 절대값 기준 축소 | 희소 행렬에 유리 | 이상치 영향 가능 |

### 2.3.3 데이터 누수는 왜 위험한가

스케일링에서 아주 중요한 개념이 **데이터 누수(data leakage)**다.  
평균과 표준편차를 계산할 때 테스트 데이터까지 같이 써 버리면, 미래 정보를 미리 본 셈이 된다.

그래서 반드시 다음 순서를 지켜야 한다.

1. 훈련 데이터로 스케일러를 `fit`
2. 훈련 데이터와 테스트 데이터를 각각 `transform`

즉, 테스트 데이터는 기준을 만드는 데 쓰면 안 되고, **이미 만들어진 기준에 맞춰 변환만** 해야 한다.

### 2.3.4 실습: 스케일링과 파이프라인

[practice/chapter02/code/2-2-scaling-pipeline.py](/Users/callii/Documents/dataScience/practice/chapter02/code/2-2-scaling-pipeline.py) 는 스케일링 방법 비교와 데이터 누수 방지를 함께 보여 준다.

실습에서는 다음을 비교한다.

pipeline = Pipeline([

이 구조의 장점은 명확하다.

- 전처리와 모델을 하나로 묶는다

3. "좋은 스케일링"보다 "모델과 맞는 스케일링"이 더 중요한가

이 실습은 학생에게 한 가지 분명한 감각을 준다.  
**전처리는 모델과 따로 노는 것이 아니라, 모델과 한 몸처럼 설계해야 한다.**

## 2.4 차원 축소: 너무 많은 축에서 핵심만 남기기

변수가 많아지면 좋은 점도 있지만, 문제도 급격히 늘어난다.

- 계산량 증가
- 과적합 위험 증가
- 시각화 어려움
- 거리 개념 왜곡

이 현상을 보통 "차원의 저주"라고 부른다.

### 2.4.1 차원의 저주를 직관적으로 이해하기

차원이 높아지면 데이터는 서로 더 멀어지고, 가까운 이웃의 의미가 흐려진다.  
즉, 2차원에서는 뚜렷하게 보이던 구조가 고차원에서는 직관적으로 파악하기 어려워진다.

쉽게 비유하면:

- 2차원 지도에서는 동네 구조가 잘 보임
- 100차원 공간에서는 동시에 너무 많은 방향을 봐야 해서 감각이 무너짐

그래서 차원 축소는 "정보를 버리는 작업"이 아니라, **핵심 구조를 남기는 압축 작업**으로 이해하는 편이 맞다.

### 2.4.2 PCA: 분산이 큰 방향을 찾는 방법

PCA는 가장 널리 쓰이는 차원 축소 방법이다.  
핵심 생각은 단순하다.

"데이터가 가장 많이 퍼져 있는 방향을 먼저 잡자."

즉, 데이터의 변화를 가장 잘 설명하는 축을 새로 만들어, 그 축들로 원래 데이터를 표현하는 것이다.

PCA는 다음 상황에 특히 유용하다.

- 빠른 차원 축소
- 전처리용 입력 압축
- 다중공선성 완화
- 대략적인 구조 확인

다만 PCA는 선형 방법이기 때문에, 복잡한 비선형 구조를 시각화하는 데는 한계가 있다.

### 2.4.3 t-SNE와 UMAP: 시각화 중심의 비선형 축소

t-SNE와 UMAP은 "눈으로 군집 구조를 보기 좋게" 만드는 데 강하다.

- **t-SNE**: 가까운 이웃 관계를 잘 보존
- **UMAP**: 지역 구조와 전역 구조를 좀 더 균형 있게 보존, 속도도 빠른 편

직관적으로 보면,

- PCA는 전체 지형을 크게 보는 지도
- t-SNE는 가까운 골목 구조를 자세히 보는 지도
- UMAP은 둘의 균형을 잡은 지도

### 2.4.4 차원 축소 방법 선택 기준

<표 2-6: 차원 축소 방법 선택 기준>

| 목적 | 적합한 방법 | 이유 |
| ---- | ----------- | ---- |
| 빠른 전처리 | PCA | 속도와 안정성이 좋음 |
| 소규모 데이터 시각화 | t-SNE | 지역 구조 표현에 강함 |
| 대규모 시각화 | UMAP | 속도와 품질 균형 |
| 후속 모델 입력 압축 | PCA 우선 | 새로운 데이터에도 적용 쉬움 |

### 2.4.5 실습: PCA, t-SNE, UMAP 비교

[practice/chapter02/code/2-3-dimensionality-reduction.py](/Users/callii/Documents/dataScience/practice/chapter02/code/2-3-dimensionality-reduction.py) 는 손글씨 숫자 데이터를 이용해 PCA, t-SNE, UMAP을 비교한다.

이 실습에서 봐야 할 포인트는 세 가지다.

1. 처리 시간
2. 군집 분리도
3. 시각화 결과의 해석 가능성

코드 흐름은 다음과 같이 정리된다.

```python
X_pca, time_pca, explained_var = reduce_with_pca(X_scaled)
X_tsne, time_tsne = reduce_with_tsne(X_scaled)
X_umap, time_umap = reduce_with_umap(X_scaled)
```

이 실습의 교육적 핵심은 "무조건 예쁜 그림이 좋은 방법은 아니다"는 점이다.

- PCA는 빠르고 재현성 좋음
- t-SNE는 보기 좋지만 느리고 설정에 민감
- UMAP은 시각화 품질과 속도의 균형이 좋음

즉, 차원 축소 방법은 **보는 목적과 쓰는 목적을 구분해서 선택**해야 한다.

## 2.5 특성 공학: 모델이 배우기 쉬운 표현 만들기

특성 공학은 원시 데이터를 모델이 더 잘 배울 수 있는 형태로 바꾸는 작업이다.  
단순히 열을 더 만드는 작업이 아니라, **데이터를 표현하는 방식을 설계하는 작업**에 가깝다.

### 2.5.1 왜 특성 공학이 필요한가

같은 정보라도 어떻게 표현하느냐에 따라 모델의 성능이 크게 달라진다.

예를 들어,

- 가격은 로그 변환을 해야 패턴이 더 잘 드러날 수 있다
- 날짜는 월, 요일, 주말 여부로 쪼개야 의미가 살아난다
- 두 변수의 곱이나 비율이 원래 변수보다 더 중요할 수 있다

즉, 특성 공학은 데이터를 "그대로 쓰는 것"이 아니라, **모델이 이해하기 쉬운 언어로 번역하는 일**이다.

### 2.5.2 자주 쓰는 특성 공학 예시

<표 2-7: 자주 쓰는 특성 공학 예시>

| 원시 데이터 | 파생 특성 예시 | 이유 |
| ----------- | -------------- | ---- |
| 소득 | 로그 소득 | 치우친 분포 완화 |
| 날짜 | 요일, 월, 주말 여부 | 반복 패턴 반영 |
| 거래 내역 | 최근 7일 거래 수 | 행동 패턴 요약 |
| 위치 정보 | 이전 위치와의 거리 | 이동 이상 탐지 |
| 두 변수 조합 | 면적당 가격, 클릭률 | 관계를 직접 표현 |

### 2.5.3 LLM은 특성 아이디어 생성에 도움을 준다

최근에는 LLM을 이용해 파생 특성 아이디어를 빠르게 브레인스토밍할 수 있다.

예를 들어 신용카드 사기 탐지에서 LLM은 다음과 같은 아이디어를 제안할 수 있다.

- 최근 24시간 내 거래 횟수
- 평소 거래 금액 대비 비율
- 직전 거래 위치와의 거리
- 야간 거래 여부

하지만 여기서 중요한 점은 분명하다.  
LLM은 아이디어를 줄 수는 있어도, 그 아이디어가 실제로 유효한지는 검증해야 한다.

즉, 특성 공학에서 LLM의 역할은 **대체자**가 아니라 **보조 아이디어 생성기**다.

## 2.6 텍스트 임베딩: 문장을 숫자로 바꾸되 의미를 잃지 않기

텍스트는 머신러닝 모델에 바로 들어갈 수 없다. 결국 숫자로 바꿔야 한다.  
문제는 어떻게 바꾸느냐이다.

### 2.6.1 TF-IDF: 전통적인 텍스트 수치화

TF-IDF는 단어가 문서 안에서 얼마나 자주 나오고, 전체 문서에서는 얼마나 드문지를 반영해 가중치를 주는 방법이다.

장점은 분명하다.

- 빠르다
- 구현이 쉽다
- 기준선으로 좋다

하지만 한계도 뚜렷하다.

- 의미 유사성을 잘 담지 못함
- 문맥을 잘 반영하지 못함
- 차원이 커지기 쉬움

즉, TF-IDF는 "단어 빈도 중심 표현"이다.

### 2.6.2 임베딩: 의미를 보존하는 벡터 표현

임베딩은 문장이나 단어를 밀집 벡터로 바꾼다.  
의미가 비슷한 문장끼리는 벡터 공간에서 가깝게 놓이도록 학습된다.

쉽게 말하면,

- TF-IDF는 단어 개수표에 가깝고
- 임베딩은 의미 지도를 만드는 방법에 가깝다

그래서 임베딩은 다음과 잘 맞는다.

- 유사 문서 검색
- 의미 기반 군집화
- 적은 라벨 데이터에서의 분류
- RAG 검색

### 2.6.3 실습: TF-IDF와 SBERT 비교

[practice/chapter02/code/2-5-text-embedding.py](/Users/callii/Documents/dataScience/practice/chapter02/code/2-5-text-embedding.py) 는 TF-IDF와 Sentence-BERT 임베딩을 비교하는 실습이다.

핵심 흐름은 다음과 같다.

```python
X_train_tfidf, X_test_tfidf, time_tfidf = vectorize_tfidf(texts_train, texts_test)
X_train_sbert, X_test_sbert, time_sbert = vectorize_sbert(texts_train, texts_test)
```

그리고 같은 분류기인 로지스틱 회귀를 써서 비교한다.

```python
result_tfidf = train_and_evaluate(X_train_tfidf, X_test_tfidf, y_train, y_test, "TF-IDF")
result_sbert = train_and_evaluate(X_train_sbert, X_test_sbert, y_train, y_test, "SBERT")
```

이 실습에서 확인할 포인트는 다음과 같다.

1. 정확도 차이
2. 벡터 차이(희소/밀집)
3. 차원 수 차이
4. 처리 시간 차이

즉, 임베딩은 "더 똑똑한 숫자 표현"이지만, 비용과 시간도 같이 본다는 관점이 중요하다.

### 2.6.4 범주형 변수 임베딩까지의 확장

임베딩 개념은 텍스트에만 쓰이지 않는다. 범주형 변수도 임베딩으로 표현할 수 있다.

예를 들어 도시 변수를 생각해 보자.

- 원-핫 인코딩에서는 서울, 부산, 인천이 완전히 별개다
- 임베딩에서는 서울과 인천이 부산보다 가까울 수 있다

즉, 범주 사이의 관계를 학습해 표현할 수 있다.  
이 관점은 이후 딥러닝, 추천 시스템, 그래프 학습으로 자연스럽게 이어진다.
