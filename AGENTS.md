# AGENTS.md

이 파일은 Codex/ChatGPT 계열 에이전트가 `/Users/callii/Documents/dataScience` 저장소에서 작업할 때 따를 운영 규칙을 정의한다.

## 1) 기본 원칙

- 이 저장소의 목표는 `docs/` 원고를 바탕으로 `lecture/`에 **학부생도 읽을 수 있는 강의자료 완성 원고**를 만드는 것이다.
- `docs/`는 소스 원고다. `lecture/`는 단순 복사본이 아니라 **재구성된 최종 강의자료**다.
- 문서는 **교수와 학생이 함께 보는 자료**다. 메타 발화나 수업 진행용 잡담 표현을 넣지 않는다.
- 강의자료는 초안 수준이 아니라, 가능한 한 **바로 수업에 사용할 수 있는 완성 원고** 수준으로 작성한다.

## 2) 우선 참조 순서

1. 사용자의 현재 요청
2. 이 파일 `AGENTS.md`
3. `lecture/STYLE_GUIDE.md`
4. `lecture/TEMPLATE.md`
5. 기존 강의자료 참고 기준: `/Users/callii/Documents/causal/lecture/`
6. 현재 프로젝트의 소스 원고: `docs/ch01.md` ~ `docs/ch12.md`
7. 실습 코드와 데이터: `practice/`
8. 그림/시각 자료: `diagram/`, `content/graphics/`

## 3) 저장소 구조 규칙

- `docs/`: 기존 원고 보관 및 참고용
- `lecture/`: 최종 강의자료
- `practice/`: 장별 실습 코드와 데이터
- `diagram/`: 문서 삽입용 그림 파일
- `content/graphics/`: 보조 시각 자료 및 HTML/PNG 자산

장 번호는 반드시 아래 형식을 유지한다.

- 문서: `lecture/chapter01.md` ~ `lecture/chapter12.md`
- 실습: `practice/chapter01/` ~ `practice/chapter12/`

## 4) 집필 규칙

- `lecture/` 문서는 `causal/lecture` 스타일을 따른다.
- 문체는 쉽고 직관적이어야 하며, 필요하면 비유와 사례를 사용한다.
- 단, 불필요하게 가벼운 표현이나 메타 표현은 넣지 않는다.
- `학생에게 남겨야 할 메시지`, `수업에서 이렇게 말하면 된다`, `여기서 멈춰라` 같은 표현은 사용하지 않는다.
- 교수-학생 공용 자료이므로 설명형 제목과 중립적인 문장으로 쓴다.

## 5) 강의자료 구성 규칙

각 장은 기본적으로 다음 요소를 포함한다.

1. 장 제목
2. 학습 목표
3. 이 장에서 배울 내용 미리보기
4. 본문 절 구조 (`N.1`, `N.2`, `N.3` ...)
5. 쉬운 설명, 예시, 비교표
6. 필요한 그림 배치
7. 실습 코드와 직접 연결되는 설명
8. 장 전체 핵심 요약

중요:
- **이론 파트와 실습 파트를 분리하지 않는다.**
- `docs/`와 마찬가지로, **방법 설명 바로 뒤에 예시와 실습 해석을 붙인다.**
- 강의 흐름은 `개념 -> 예시 -> 실습 코드/결과 해석 -> 다음 개념` 순서를 기본으로 한다.
- 앞으로 모든 장의 실제 작성 기준은 **`lecture/chapter03.md`를 직접 참조**한다.
- 새 장을 작성하거나 기존 장을 확장할 때는 `lecture/chapter03.md`의 다음 요소를 맞춘다.
  - 설명 밀도
  - 표 배치 방식
  - 그림 배치 방식
  - 개념과 실습을 섞는 방식
  - 메타 표현 없이 설명형 제목으로 정리하는 방식

## 6) 분량 기준

- 3시간 수업 기준 장은 짧은 개요 수준으로 끝내지 않는다.
- 줄 수보다 중요한 것은 **설명 밀도**다.
- 완성 원고는 보통 다음을 충분히 포함해야 한다.
  - 개념 정의
  - 쉬운 비유
  - 숫자 예시 또는 작은 표
  - 그림
  - 실습 코드 연결
  - 실습 결과 해석
  - 비교/선택 기준

즉, `lecture/`는 "요약본"이 아니라 **강의용 본문 원고**다.

## 7) 실습 연결 규칙

- 관련된 모든 실습은 문서에 반드시 포함한다.
- 실습은 장 끝에 몰아넣지 않고, 관련 개념 직후에 배치한다.
- 실습 경로는 반드시 실제 경로를 쓴다.
  - 예: `practice/chapter03/code/3-3-titanic-decision-tree.py`
- 실습을 언급할 때는 아래를 함께 설명한다.
  - 이 실습이 어떤 개념을 보여 주는가
  - 코드에서 무엇을 봐야 하는가
  - 결과를 어떻게 읽어야 하는가

## 8) 그림/자산 사용 규칙

- 문서 작성 시 `diagram/`과 `content/graphics/`를 반드시 확인한다.
- 기존에 있는 적절한 그림이 있으면 직접 문서에 배치한다.
- 그림이 없으면 먼저 기존 자산으로 대체 가능한지 확인한다.
- `docs/`가 이미 참조하는 그림이 있으면 가능한 한 같은 그림 체계를 유지한다.
- 현재 저장소의 기본 그림 경로는 `../diagram/...` 형식이다.

## 9) 문체 금지 규칙

다음 표현은 사용하지 않는다.

- 학생에게 남겨야 할 메시지
- 수업에서 이렇게 말하면 된다
- 여기서 멈춰야 한다
- 이번 실습은...
- 바로 던질 질문
- 수업 운영용 메타 멘트 전반

대신 아래처럼 쓴다.

- 해석의 핵심
- 결과 해석
- 확인 질문
- 예시
- 비교 기준

## 10) 품질 규칙

- 설명은 쉬워야 하지만 얕아서는 안 된다.
- `docs/`의 내용을 무비판적으로 줄이지 않는다.
- 중요한 개념, 표, 수치 예시, 비교 기준은 유지하거나 강의용으로 재구성한다.
- 가능하면 그림과 표를 활용해 읽기 부담을 낮춘다.
- 실습 코드와 문서 내용이 어긋나지 않도록 확인한다.
- 문서에서 참조하는 파일 경로는 실제 존재해야 한다.

## 11) 저장소 정리 규칙

- 임시 산출물은 남기지 않는다.
- 아래 항목은 버전 관리 대상이 아니다.
  - `.DS_Store`
  - `tmpclaude-*`
  - 장별 `venv/`
  - 기타 임시 캐시

## 12) 작업 우선순위

별도 지시가 없으면 아래 순서를 따른다.

1. 저장소 구조와 임시 산출물 정리
2. `lecture/` 문서의 구조와 스타일 정비
3. 개별 장을 **완성 원고 수준**으로 작성
4. 관련 그림 배치
5. 실습 연결 및 해석 보강
6. 문체 정리 및 메타 표현 제거

- `docs/preface.md`, `docs/references.md`, `docs/bio.md`는 사용하지 않는다.
- `lecture/`는 이미 생성되어 있으며, `chapter01.md` ~ `chapter12.md`를 유지한다.
- `practice/` 장 번호는 `chapter01` 형식으로 통일되어 있다.
- 현재 작성 기준은 `/Users/callii/Documents/causal/lecture/`의 서술 방식이다.
- `lecture/chapter03.md`는 이론과 실습을 섞는 방식으로 확장 중이며, 이 방식이 다른 장에도 적용되어야 한다.
- **앞으로 모든 장의 집필은 `lecture/chapter03.md`를 직접 참조하는 것을 기본 규칙으로 한다.**
